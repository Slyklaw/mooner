## Directory Purposes

**Root Directory:**
- Purpose: Core compiler implementation (all stages in 4 files)
- Contains: `lexer.mbt`, `parser.mbt`, `codegen.mbt`, `compiler.mbt`, `double_ryu_nonjs.mbt`
- Also contains: `moon.mod.json` (project metadata), `moon.pkg` (dependencies), test files
- Notable: Unusual flat structure - no `src/` directory

**cmd/main/:**
- Purpose: Command-line interface
- Contains: `main.mbt` (24 lines)
- Responsibilities: Argument parsing, output path resolution, invoke compiler, print results
- Build output: `_build/native/debug/cmd/main`

**examples/:**
- Purpose: Test programs and examples
- Contains: 70+ MoonBit source files
- `mbt_examples/`: Official MoonBit test suite used for compiler validation
- Simple programs: `hello.mbt`, `simple.mbt`, `empty.mbt`, `just_int.mbt`
- Used by `run_e2e_tests.sh` to compare against reference compiler

**.mooncakes/:**
- Purpose: MoonBit package cache (downloaded dependencies)
- Generated by: `moon build`
- Not committed to git
- Contains: `moonbitlang/x/` packages (standard library)

**_build/:**
- Purpose: Build artifacts and intermediates
- Generated by: `moon build`
- Not committed to git
- Contains: native and wasm build outputs

## Key File Locations

**Entry Points:**
- `cmd/main/main.mbt` - CLI entry (fn main)
- `compiler.mbt:compile_file` - Core compilation entry
- `parser.mbt:parse` - Parsing entry
- `lexer.mbt:tokenize` - Lexical analysis entry
- `codegen.mbt:codegen` - Code generation entry (implied)

**Configuration:**
- `moon.mod.json` - Project metadata (name, version, dependencies, imports)
- `moon.pkg` - Package dependencies (imports `moonbitlang/x/fs`)
- `AGENTS.md` - Agent instructions (this project's coding conventions)

**Core Compiler:**
- `lexer.mbt` - Lexical analyzer (Token enum, Lexer struct)
- `parser.mbt` - Parser (AST enum, Parser struct, X86Inst/X86Operand enums)
- `codegen.mbt` - Code generator (CodeGen struct, instruction emission)
- `compiler.mbt` - Orchestration (ELF header generation, file I/O)

**Utilities:**
- `double_ryu_nonjs.mbt` - Float-to-string conversion algorithm (Ryu)

**Tests:**
- `mooner_test.mbt` - Blackbox E2E tests (API validation)
- `mooner_wbtest.mbt` - Whitebox unit tests (lexer, parser internals)
- `examples/mbt_examples/*.mbt` - Integration tests against reference compiler
- `run_e2e_tests.sh` - Bash script for E2E comparison

## Naming Conventions

**Files:**
- Core modules: lowercase short names (`lexer`, `parser`, `codegen`, `compiler`)
- Test files: `{module}_test.mbt` (blackbox) or `{module}_wbtest.mbt` (whitebox)
- Example programs: lowercase with optional directory prefix

**Types:**
- Enums: PascalCase (Token, AST, X86Inst, X86Operand)
- Structs: PascalCase (Lexer, Parser, CodeGen)
- Variants: PascalCase (Fn, Let, Int, Float, Mov, Add, etc.)

**Functions:**
- snake_case for functions and methods
- Constructor functions: `Type::new` (Lexer::new, Parser::new, CodeGen::new)
- Parser combinators: `parse_*` (parse_ident, parse_expr, parse_block)
- CodeGen emitters: `emit_*` (emit_byte, emit32, emit64, emit_inst)
- Public API: lowercase (tokenize, parse, codegen, compile_file)

**Variables:**
- snake_case for local variables and fields
- Immutable by default (MoonBit convention)
- Mutables: `mut` keyword (rare: `mut lexer`, `mut parser`, `mut result`)

**Constants:**
- PascalCase for global constants (rare) or all-caps for byte values
- ELF magic bytes: `0x7F, 0x45, 0x4C, 0x46`

## Where to Add New Code

**New Compiler Feature (e.g., new AST node):**
1. Add variant to `AST` enum in `parser.mbt` (~line 40)
2. Add parsing logic in appropriate `parse_*` function in `parser.mbt`
3. Add code generation in `codegen.mbt` (match arm in `codegen` or helper)
4. Add tests:
   - Unit test in `mooner_wbtest.mbt` for parsing
   - E2E test in `examples/mbt_examples/XXX_new_feature.mbt`

**New Optimization:**
- Add transformation pass in `parser.mbt` or create new `optimize` function
- Or integrate directly into `codegen` emission logic

**New x86 Instruction:**
1. Add variant to `X86Inst` enum in `parser.mbt` (~line 1641)
2. Add match arm in `CodeGen::emit_inst` in `codegen.mbt` (~line 376)
3. Implement byte encoding patterns following existing style

**New Standard Library Function:**
- Not handled here - this compiler emits direct syscalls
- Runtime support would require new AST node and codegen logic

**CLI Enhancement:**
- Modify `cmd/main/main.mbt`
- Keep it minimal - complex logic belongs in `compiler.mbt`

## Special Directories

**`.mooncakes/`:**
- Purpose: MoonBit package cache (similar to npm/node_modules)
- Generated: Yes (by `moon build`)
- Committed: No (in .gitignore)

**`_build/`:**
- Purpose: Compiler build artifacts (native and wasm targets)
- Generated: Yes
- Committed: No (in .gitignore)

**`examples/mbt_examples/`:**
- Purpose: Official MoonBit test suite
- Generated: No (committed)
- These are golden files - outputs must match reference compiler exactly

## Module Boundaries

**No package boundaries** - all code lives in single package (implicit root package).

**Internal visibility:**
- `fn` keyword: public (exported)
- No modifier: private to package (MoonBit default)
- `priv` keyword: truly private (within file)
- Most helper functions are private (e.g., `Lexer::current_char`, `Parser::_peek`)

**Test visibility:**
- Tests in `_test.mbt` and `_wbtest.mbt` can access private symbols of same package
- No test-specific exports needed

## Build Structure

**Build command:** `moon build`
- Produces: `_build/native/debug/cmd/main` (compiler executable)

**Run command:** `moon run cmd/main <input> [output]`
- Executes the built compiler with arguments

**Test command:** `moon test`
- Runs blackbox and whitebox tests

**Format command:** `moon fmt`
- Formats all `.mbt` files according to MoonBit style

**Info command:** `moon info`
- Generates `.mbti` interface files for package metadata
